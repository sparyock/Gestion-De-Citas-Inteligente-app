# 1. Identificación de Patrones

## 1.1 Patrones Actualmente Implementados

En el sistema actual se utilizan excepciones genéricas para manejar errores.

| Patrón | Ubicación | Descripción |
|--------|-----------|------------|
| Exception Handling | TurnoService | Manejo básico de errores |
| REST API Pattern | TurnoController | Respuesta HTTP a errores |
| Service Layer | TurnoService | Control de lógica de negocio |

---

## 1.2 Patrones Ausentes Críticos

| Patrón Faltante | Justificación |
|----------------|--------------|
| Custom Exception Pattern | No existe manejo específico por dominio |
| Error Handling Strategy | Falta control centralizado de errores |
| Domain Exception | No hay excepciones propias del negocio |

# 3. Identificación de Antipatrones

## Antipatrón: Uso de RuntimeException

Ejemplo:

```java
throw new RuntimeException("Turno no encontrado");
```

Problemas:

- No identifica el tipo de error  
- No permite control específico  
- Dificulta mantenimiento  

---

# 4. Segmentos de Código con Malas Prácticas

```java
if(turno == null){
    throw new RuntimeException("Turno no encontrado");
}
```

Problema:

Uso de excepción genérica.

# 5. Diseño e Implementación del ADR

## 5.1 Contexto

El sistema requiere un manejo adecuado de errores en:

- Turnos  
- Usuarios  
- Doctores  

---

## 5.2 Decisión

Implementar excepciones personalizadas por dominio.

---

## 5.3 Diseño de la Solución

```
Controller
   │
   ▼

Service
   │
   ▼

Custom Exception
```

---

## 5.4 Implementación

```java
public class ResourceNotFoundException extends RuntimeException {

    public ResourceNotFoundException(String message){
        super(message);
    }

}
```

Uso:

```java
throw new ResourceNotFoundException(
    "Turno no encontrado con id: " + id
);

```
---

## 5.5 Estructura de Paquetes

```
exception
   │
   └── ResourceNotFoundException
```

# 6. Impacto sobre el Sistema

## 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|------------|--------|
Mejor control de errores | Identificación clara | Alto
Mayor mantenibilidad | Código organizado | Alto
Escalabilidad | Nuevas excepciones | Medio

---

## 6.2 Métricas Esperadas

ANTES

RuntimeException  

DESPUÉS

ResourceNotFoundException  
ValidationException  

---

## 6.3 Consecuencias

Positivas:

- Mejor manejo de errores  
- Código más claro  
- Mejor mantenimiento  

Negativas:

- Requiere crear nuevas clases  

---

# Decisión Final

Se implementarán excepciones personalizadas por dominio para mejorar el manejo de errores en el sistema de Gestión de Citas Inteligente.
