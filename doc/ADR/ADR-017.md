## 2. Análisis del Proyecto

### 2.1 Estado Actual de la Arquitectura

El API Gateway solo configura CORS y enruta peticiones. No existe ningún mecanismo de autenticación. Cualquier cliente puede consumir `/users`, `/turnos` u otros microservicios sin presentar credenciales.
```
┌─────────────────────────────────────────────┐
│            ARQUITECTURA ACTUAL              │
├─────────────────────────────────────────────┤

Cliente (cualquiera)
        │
        ▼
  API Gateway :8080
  CorsConfig → allowedOrigin("*")  ← SIN FILTRO
        │
        ├──→ users-service   :8081  ← acceso libre
        └──→ turnos-service  :8082  ← acceso libre

└─────────────────────────────────────────────┘
```

**Problema detectado:** Ausencia total de capa de seguridad. El gateway expone todos los microservicios sin control de acceso.

### 2.2 ADR Seleccionado

| ADR ID | Título | Prioridad | Razón |
|--------|--------|-----------|-------|
| ADR-017 | Proxy: Filtro de autenticación en Gateway |  Alta | El sistema no tiene ningún control de acceso entre cliente y microservicios |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Absent Security Layer

**Ubicación:** `api-gateway/CorsConfig.java`
```java
config.addAllowedOrigin("*");  // ← cualquier origen del mundo
config.addAllowedMethod("*");  // ← cualquier método HTTP
config.addAllowedHeader("*");  // ← cualquier cabecera
// No existe ningún filtro de autenticación
```

**Problemas:**
- Cualquier cliente anónimo puede acceder a datos médicos
- No hay diferencia entre un usuario autenticado y uno no autenticado
- El sistema es vulnerable a accesos no autorizados

### 3.2 Antipatrón: Open CORS

**Problemas:**
- `allowedOrigin("*")` en producción expone el sistema a peticiones desde cualquier dominio
- No se restringe a los orígenes legítimos (ej. frontend Angular en `localhost:4200`)

```java
// CorsConfig.java — ANTES
@Configuration
public class CorsConfig {
    @Bean
    public CorsWebFilter corsWebFilter() {
        CorsConfiguration config = new CorsConfiguration();
        config.addAllowedOrigin("*");  // ← cualquier origen
        config.addAllowedMethod("*");  // ← cualquier método HTTP
        config.addAllowedHeader("*");  // ← cualquier cabecera
        // No existe ningún filtro de autenticación / autorización
        SourceBasedCorsConfigurationSource source =
            new SourceBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);
        return new CorsWebFilter(source);
    }
}
```

**Problemas:**
- Wildcard en origen, método y cabecera simultáneamente
- Sin filtro que valide tokens Bearer
- No distingue rutas públicas de rutas protegidas

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El API Gateway es el único punto de entrada al sistema. Centralizar la autenticación aquí garantiza que ningún microservicio sea accesible sin credenciales válidas, sin necesidad de implementar seguridad en cada uno de ellos.

### 5.2 Decisión

Se decide implementar el **patrón Proxy** mediante un `GlobalFilter` de Spring Cloud Gateway. El filtro intercepta todas las peticiones, verifica el token Bearer y solo delega al microservicio real si la autenticación es válida.

| Componente | Responsabilidad |
|------------|-----------------|
| `AuthorizationFilter` | Proxy que intercepta y valida todas las peticiones |
| `RUTAS_PUBLICAS` | Lista de rutas que no requieren token (login, registro) |
| `CorsConfig` | Restringido al origen del frontend Angular |

### 5.3 Diseño de la Solución
```
┌────────────────────────────────────────────┐
│           ARQUITECTURA PROPUESTA           │
├────────────────────────────────────────────┤

Cliente
    │
    ▼
API Gateway :8080
    │
    ▼
AuthorizationFilter (Proxy)
    ├── ¿ruta pública? → pasa directo
    ├── ¿sin token?    → 401 Unauthorized
    └── ¿token válido? → delega al microservicio
              │
              ├──→ users-service  :8081
              └──→ turnos-service :8082

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

**Proxy: AuthorizationFilter**
```java
// AuthorizationFilter.java
@Component
public class AuthorizationFilter implements GlobalFilter, Ordered {

    private static final List<String> RUTAS_PUBLICAS =
        List.of("/users/login", "/users");

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 1. Rutas públicas pasan directamente al microservicio real
        boolean esPublica = RUTAS_PUBLICAS.stream()
            .anyMatch(ruta -> path.equals(ruta) ||
                (path.equals("/users") &&
                 exchange.getRequest().getMethod().name().equals("POST")));

        if (esPublica) return chain.filter(exchange);

        // 2. Verificar cabecera Authorization: Bearer <token>
        String authHeader = exchange.getRequest()
            .getHeaders().getFirst(HttpHeaders.AUTHORIZATION);

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete(); // corta la cadena
        }

        // 3. Delegar al microservicio real (objeto proxeado)
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() { return -1; } // se ejecuta antes que cualquier otro filtro
}
```

**CorsConfig corregido**
```java
// CorsConfig.java — CORREGIDO
config.addAllowedOrigin("http://localhost:4200"); // solo el frontend Angular
config.addAllowedMethod("GET");
config.addAllowedMethod("POST");
config.addAllowedMethod("PUT");
config.addAllowedMethod("DELETE");
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|-------------|---------|
| Seguridad centralizada | Autenticación en un solo lugar, no en cada microservicio | Alto |
| Transparencia | Los microservicios no saben que existe el proxy | Alto |
| Single Point of Control | Cambiar política de auth = editar solo AuthorizationFilter | Alto |
| CORS restringido | Solo el frontend Angular puede consumir el gateway | Medio |

### 6.2 Métricas Esperadas
```
ANTES
CorsConfig.java         ~20 líneas (sin seguridad)
Microservicios          Acceso público sin restricción

DESPUÉS
AuthorizationFilter.java  ~40 líneas (proxy de seguridad)
CorsConfig.java           ~15 líneas (restringido)
Microservicios            Solo accesibles con token válido
```

### 6.3 Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Usuarios sin token rechazados | Media | Medio | Lista de rutas públicas bien definida |
| Token no emitido por users-service | Media | Alto | users-service ya tiene jjwt en pom.xml |

### 6.4 Plan de Implementación

**Fase 1:**
- Crear `AuthorizationFilter` con lista de rutas públicas

**Fase 2:**
- Corregir `CorsConfig` para restringir origen al frontend

**Fase 3:**
- Verificar que `users-service` emite tokens JWT en login

---

 **Decisión Final**

Se aprueba la implementación del patrón Proxy mediante `GlobalFilter` en el API Gateway para interceptar y validar todas las peticiones antes de que lleguen a los microservicios, garantizando que solo usuarios autenticados accedan al sistema.
