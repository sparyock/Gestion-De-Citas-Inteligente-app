## 3. Identificación de Antipatrones

### 3.1 Antipatrón: God Class / Mixed Responsibilities

**Ubicación:** `UserController.java`

```java
@RestController
@RequestMapping("/users")
public class UserController {
    // CRUD de usuarios
    @PostMapping          // crear
    @GetMapping           // listar
    @PutMapping("/{id}")  // actualizar
    @DeleteMapping("/{id}") // eliminar

    // ← AUTENTICACIÓN mezclada con CRUD
    @PostMapping("/login")
    public ResponseEntity<UserResponseDTO> login(...) { }
}
```

**Problema:**

- Una sola clase tiene dos responsabilidades distintas
- El login y el CRUD son conceptos completamente diferentes
- Viola el principio de responsabilidad única (SRP)

---

### 3.2 Antipatrón: Inconsistent URL Design

**Ubicación:** `UserController.java`

```java
// ❌ Login mezclado bajo /users
@PostMapping("/login")  // → POST /users/login
```

**Problemas:**

- La URL `/users/login` no es semántica para autenticación
- El estándar de la industria es `/auth/login`
- Dificulta agregar JWT o refresh tokens en el futuro

---

## 4. Segmentos de Código con Malas Prácticas

**Login mezclado en el controlador de usuarios**

```java
// ❌ ANTES — login dentro de UserController
@PostMapping("/login")
public ResponseEntity<UserResponseDTO> login(
        @Valid @RequestBody UserRequestDTO request) {

    return ResponseEntity.ok(
            userService.login(request.getEmail(), request.getPassword())
    );
}
```

**Problemas:**

- Mezcla autenticación con gestión de usuarios
- Si se agrega JWT, hay que modificar el controlador de usuarios
- No sigue la convención REST estándar para autenticación

---

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El `UserController` maneja tanto el CRUD de usuarios como el proceso de autenticación. Estas son dos responsabilidades distintas que deben separarse en controladores independientes.

---

### 5.2 Decisión

Se decide crear un `AuthController` separado que maneje exclusivamente la autenticación bajo la ruta `/auth`. El `UserController` queda limpio con solo operaciones CRUD.

| Controlador    | Ruta base | Responsabilidad          |
| -------------- | --------- | ------------------------ |
| UserController | /users    | CRUD de usuarios         |
| AuthController | /auth     | Autenticación y login    |

---

### 5.3 Diseño de la Solución

```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

UserController (/users)
├── GET    /users
├── GET    /users/{id}
├── POST   /users
├── PUT    /users/{id}
└── DELETE /users/{id}

AuthController (/auth)
└── POST   /auth/login  ← autenticación separada

└────────────────────────────────────────────┘
```

---

### 5.4 Implementación Propuesta

**Nuevo AuthController**

```java
@RestController
@RequestMapping("/auth")
public class AuthController {

    @Autowired
    private UserService userService;

    // ADR-013: login separado del CRUD de usuarios
    @PostMapping("/login")
    public ResponseEntity<UserResponseDTO> login(
            @Valid @RequestBody UserRequestDTO request) {

        return ResponseEntity.ok(
            userService.login(request.getEmail(), request.getPassword())
        );
    }
}
```

**UserController sin login**

```java
// ✅ DESPUÉS — UserController solo maneja CRUD
@RestController
@RequestMapping("/users")
public class UserController {
    // GET, POST, PUT, DELETE únicamente
    // ADR-013: login movido a AuthController (/auth/login)
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio                 | Descripción                                              | Impacto |
| ------------------------- | -------------------------------------------------------- | ------- |
| Responsabilidad única     | Cada controlador tiene una sola responsabilidad          | Alto    |
| URL semántica             | /auth/login es el estándar de la industria               | Medio   |
| Extensibilidad            | Fácil agregar /auth/logout, /auth/refresh en el futuro   | Alto    |
| Código más limpio         | Controladores más cortos y fáciles de leer               | Medio   |

---

### 6.2 Métricas Esperadas

```
ANTES

UserController → 6 endpoints (CRUD + login)
Líneas de código en UserController → ~80 líneas


DESPUÉS

UserController → 5 endpoints (solo CRUD)
AuthController → 1 endpoint (solo login)
Líneas por controlador → ~50 líneas cada uno
```

- Reducción de responsabilidades por clase: De 2 a 1
- Mejora en organización del código: Alta

---

### 6.3 Riesgos

| Riesgo                        | Probabilidad | Impacto | Mitigación                                      |
| ----------------------------- | ------------ | ------- | ----------------------------------------------- |
| Romper clientes que usan /users/login | Media | Bajo  | Actualizar frontend a /auth/login               |
| Olvidar actualizar el Gateway | Baja         | Medio   | Agregar ruta /auth/** en application.yml        |

---

### 6.4 Plan de Implementación

**Fase 1:**
- Crear AuthController con el endpoint /auth/login

**Fase 2:**
- Eliminar el endpoint /login de UserController

**Fase 3:**
- Actualizar API Gateway para enrutar /auth/** al users-service
- Actualizar el frontend de /users/login a /auth/login

---

✅ **Decisión Final**

Se aprueba la separación del controlador de autenticación en un `AuthController` independiente, mejorando la organización del código y siguiendo el principio de responsabilidad única.
