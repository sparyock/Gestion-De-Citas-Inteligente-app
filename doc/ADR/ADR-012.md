## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado Actual de la Arquitectura

El flujo de manejo de contraseñas actualmente es:

```
┌─────────────────────────────────────────────┐
│              FLUJO ACTUAL (INSEGURO)        │
├─────────────────────────────────────────────┤

Cliente → envía password en texto plano
              │
              ▼
UserService → guarda password sin cifrar
              │
              ▼
PostgreSQL → almacena "admin1234" (legible)

              │
              ▼
Login → compara "admin1234".equals("admin1234")

└─────────────────────────────────────────────┘
```

**Problema detectado:**

Las contraseñas se almacenan en texto plano en la base de datos. Si la BD es comprometida, todas las contraseñas quedan expuestas inmediatamente.

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Plain Text Password Storage

**Ubicación:** `UserService.java`

```java
user.setPassword(request.getPassword()); // se guarda sin cifrar
```

**Problema:**

- Contraseñas visibles directamente en la base de datos
- Vulnerabilidad crítica de seguridad
- Incumple estándares básicos de seguridad (OWASP)

---

### 3.2 Antipatrón: Plain Text Password Comparison

**Ubicación:** `UserService.java`

```java
if (!user.getPassword().equals(password)) {
    throw new RuntimeException("Credenciales inválidas");
}
```

**Problemas:**

- Comparación directa de texto plano
- Si alguien accede a la BD, obtiene todas las contraseñas
- No tiene protección contra ataques de fuerza bruta

---

## 4. Segmentos de Código con Malas Prácticas

**Almacenamiento sin cifrar**

```java
// ❌ ANTES — contraseña guardada en texto plano
public UserResponseDTO create(UserRequestDTO request) {
    User user = new User();
    user.setNombre(request.getNombre());
    user.setEmail(request.getEmail());
    user.setPassword(request.getPassword()); // ← texto plano
    User saved = userRepository.save(user);
    return new UserResponseDTO(saved.getId(), saved.getNombre(), saved.getEmail());
}
```

**Problemas:**

- La contraseña "admin1234" se guarda como "admin1234" en PostgreSQL
- Cualquier DBA puede ver todas las contraseñas
- Viola el principio de mínima exposición de datos

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El sistema almacena contraseñas de usuarios para el proceso de autenticación. Actualmente no existe ningún mecanismo de cifrado, lo que representa una vulnerabilidad crítica de seguridad.

---

### 5.2 Decisión

Se decide implementar BCryptPasswordEncoder, el estándar recomendado por Spring Security, para cifrar contraseñas antes de almacenarlas y verificarlas durante el login.

| Componente           | Responsabilidad                           |
| -------------------- | ----------------------------------------- |
| BCryptPasswordEncoder | Cifra la contraseña al crear el usuario  |
| matches()            | Verifica la contraseña en el login        |
| SecurityConfig       | Expone el bean PasswordEncoder            |

---

### 5.3 Diseño de la Solución

```
┌────────────────────────────────────────────┐
│            FLUJO PROPUESTO (SEGURO)        │
├────────────────────────────────────────────┤

Cliente → envía password en texto plano
              │
              ▼
UserService → passwordEncoder.encode(password)
              │
              ▼
PostgreSQL → almacena "$2a$10$xyz..." (hash BCrypt)

              │
              ▼
Login → passwordEncoder.matches(input, hash)

└────────────────────────────────────────────┘
```

---

### 5.4 Implementación Propuesta

**Bean PasswordEncoder en SecurityConfig**

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

**Al crear usuario**

```java
// ✅ DESPUÉS — contraseña cifrada con BCrypt
user.setPassword(passwordEncoder.encode(request.getPassword()));
```

**Al hacer login**

```java
// ✅ DESPUÉS — verificación segura
if (!passwordEncoder.matches(password, user.getPassword())) {
    throw new RuntimeException("Credenciales inválidas");
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio                    | Descripción                                            | Impacto |
| ---------------------------- | ------------------------------------------------------ | ------- |
| Seguridad de contraseñas     | Las contraseñas nunca se almacenan legibles            | Alto    |
| Protección ante brechas      | Si la BD es comprometida, los hashes no son reversibles | Alto   |
| Salt automático              | BCrypt agrega salt único por contraseña               | Alto    |
| Estándar de la industria     | BCrypt es el algoritmo recomendado por OWASP           | Medio   |

---

### 6.2 Métricas Esperadas

```
ANTES

BD PostgreSQL → password: "admin1234" (texto plano visible)
Login         → "admin1234".equals("admin1234")


DESPUÉS

BD PostgreSQL → password: "$2a$10$N9qo8..." (hash irreversible)
Login         → passwordEncoder.matches("admin1234", "$2a$10$N9qo8...")
```

- Nivel de seguridad de contraseñas: Crítico → Seguro
- Impacto en rendimiento: Mínimo (BCrypt está optimizado)

---

### 6.3 Riesgos

| Riesgo                           | Probabilidad | Impacto | Mitigación                                    |
| -------------------------------- | ------------ | ------- | --------------------------------------------- |
| Contraseñas antiguas inválidas   | Alta         | Medio   | Limpiar BD al implementar (entorno desarrollo)|
| Mayor tiempo de procesamiento    | Baja         | Bajo    | BCrypt tiene costo configurable               |

---

### 6.4 Plan de Implementación

**Fase 1:**
- Agregar bean BCryptPasswordEncoder en SecurityConfig

**Fase 2:**
- Actualizar método create() para cifrar contraseña

**Fase 3:**
- Actualizar método login() para usar matches()

---

✅ **Decisión Final**

Se aprueba la implementación de BCrypt para el cifrado de contraseñas en el microservicio `users-service`, eliminando la vulnerabilidad crítica de almacenamiento en texto plano.
