## 2. Análisis del Proyecto

### 2.1 Estado Actual de la Arquitectura

El sistema no registra ninguna operación. No hay logs de creación de usuarios, cambios de estado de turnos ni cancelaciones. En producción es imposible rastrear qué operación falló ni quién la ejecutó.
```
┌─────────────────────────────────────────────┐
│            ARQUITECTURA ACTUAL              │
├─────────────────────────────────────────────┤

UserService
├── create()       ← sin ningún log
├── update()       ← sin ningún log
└── delete()       ← sin ningún log

TurnoService
├── crearTurno()   ← sin ningún log
├── cancelar()     ← sin ningún log
└── findAll()      ← sin ningún log

Producción: si falla algo → imposible saber qué, cuándo, con qué datos

└─────────────────────────────────────────────┘
```

**Problema detectado:** Ausencia total de trazabilidad. Si ocurre un error en producción no hay forma de diagnosticarlo.

### 2.2 ADR Seleccionado

| ADR ID | Título | Prioridad | Razón |
|--------|--------|-----------|-------|
| ADR-020 | Decorator: Logging en servicios |  Alta | El sistema no tiene ningún registro de operaciones críticas |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Log-less Service

**Ubicación:** `UserService.java`, `TurnoService.java`
```java
public UserResponseDTO create(UserRequestDTO request) {
    User user = new User();
    user.setNombre(request.getNombre());
    user.setEmail(request.getEmail());
    user.setPassword(request.getPassword());
    User saved = userRepository.save(user);
    return new UserResponseDTO(saved.getId(), saved.getNombre(), saved.getEmail());
    // ← ningún registro de que esta operación ocurrió
}
```

**Problemas:**
- No queda registro de quién fue creado ni cuándo
- Si falla `userRepository.save()`, no hay información del contexto
- Imposible auditar operaciones en producción

### 3.2 Antipatrón: Cross-cutting Concern incrustado

**Ubicación:** `TurnoService.java` (alternativa mala)
```java
public Turno crearTurno(TurnoRequestDTO dto) {
    System.out.println("Creando turno para " + dto.getDoctor()); // log manual sucio
    Turno t = new Turno();
    // ... setters ...
    Turno saved = turnoRepository.save(t);
    System.out.println("Turno creado ID: " + saved.getId());     // otro log manual
    return saved;
}
```

**Problemas:**
- `System.out.println()` no es configurable ni tiene niveles (INFO/ERROR/DEBUG)
- El logging mezcla responsabilidades con la lógica de negocio
- Viola el principio de Responsabilidad Única (SRP)

## 4. Segmentos de Código con Malas Prácticas

### Servicio sin trazabilidad
```java
// UserService.java — ANTES
public UserResponseDTO create(UserRequestDTO request) {
    User user = new User();
    user.setNombre(request.getNombre());
    user.setEmail(request.getEmail());
    user.setPassword(request.getPassword());
    User saved = userRepository.save(user);
    return new UserResponseDTO(saved.getId(), saved.getNombre(), saved.getEmail());
}

// TurnoService.java — ANTES
public Turno crearTurno(TurnoRequestDTO dto) {
    System.out.println("Creando turno para " + dto.getDoctor()); // log manual sucio
    Turno t = new Turno();
    // ... setters ...
    Turno saved = turnoRepository.save(t);
    System.out.println("Turno creado ID: " + saved.getId());
    return saved;
}
```

**Problemas:**
- Un servicio sin logs, el otro con `System.out.println()` sin nivel ni formato
- No hay consistencia en cómo se registran las operaciones
- Agregar logs en cada método multiplica el código y mezcla responsabilidades

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El sistema tiene múltiples servicios con múltiples métodos. Agregar logging directamente en cada método violaría SRP y generaría código repetitivo. Spring AOP permite implementar el patrón Decorator de forma transparente: el Aspect envuelve automáticamente todos los métodos de los servicios sin modificar su código fuente.

### 5.2 Decisión

Se decide implementar el **patrón Decorator** via **Spring AOP**. Un único `@Aspect` envuelve todos los métodos anotados con `@Service`, registrando inicio, resultado y duración sin tocar ninguna clase existente.

| Componente | Responsabilidad |
|------------|-----------------|
| `ServiceLoggingDecorator` | Aspect que envuelve todos los @Service automáticamente |
| `@Around` | Intercepta antes y después de cada método |
| `spring-boot-starter-aop` | Dependencia necesaria en pom.xml |

### 5.3 Diseño de la Solución
```
┌────────────────────────────────────────────┐
│           ARQUITECTURA PROPUESTA           │
├────────────────────────────────────────────┤

        ServiceLoggingDecorator (@Aspect)
               │
    ┌──────────┴──────────┐
    │                     │
    ▼                     ▼
UserService          TurnoService
  create()            crearTurno()
  update()            cancelar()
  delete()            findAll()

Cada método queda automáticamente decorado:
[INICIO] UserService.create() | args: [...]
[OK]     UserService.create() completado en 32ms
[ERROR]  TurnoService.crearTurno() lanzó: ...

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

**ServiceLoggingDecorator — Decorator via Spring AOP**
```java
// ServiceLoggingDecorator.java
@Aspect
@Component
public class ServiceLoggingDecorator {

    private static final Logger log =
        LoggerFactory.getLogger(ServiceLoggingDecorator.class);

    // Aplica a todos los métodos públicos de cualquier @Service
    @Around("execution(* com.citas..service.*Service.*(..))")
    public Object logearOperacion(ProceedingJoinPoint pjp) throws Throwable {
        String metodo   = pjp.getSignature().getName();
        String servicio = pjp.getTarget().getClass().getSimpleName();
        long inicio     = System.currentTimeMillis();

        log.info("[INICIO] {}.{}() | args: {}",
            servicio, metodo, Arrays.toString(pjp.getArgs()));

        try {
            Object resultado = pjp.proceed(); // ejecuta el método original
            long duracion = System.currentTimeMillis() - inicio;
            log.info("[OK] {}.{}() completado en {}ms", servicio, metodo, duracion);
            return resultado;
        } catch (Throwable ex) {
            log.error("[ERROR] {}.{}() lanzó: {}", servicio, metodo, ex.getMessage());
            throw ex; // re-lanza para que GlobalExceptionHandler lo maneje
        }
    }
}
```

**Dependencia en pom.xml**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

**Ejemplo de salida en consola**
```
[INICIO] TurnoService.crearTurno() | args: [TurnoRequestDTO(especialidad=Pediatría, ...)]
[OK]     TurnoService.crearTurno() completado en 43ms

[INICIO] UserService.create() | args: [UserRequestDTO(nombre=Juan, email=juan@mail.com)]
[OK]     UserService.create() completado en 28ms

[INICIO] TurnoService.cancelar() | args: [5]
[ERROR]  TurnoService.cancelar() lanzó: Turno no encontrado con id: 5
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|-------------|---------|
| SRP | UserService y TurnoService quedan sin ninguna línea de log | Alto |
| No invasivo | El Decorator se activa automáticamente en todos los @Service | Alto |
| Trazabilidad | Cada operación queda registrada con duración y parámetros | Alto |
| Diagnóstico | El log de ERROR muestra exactamente qué servicio y método falló | Alto |

### 6.2 Métricas Esperadas
```
ANTES
UserService.java        ~60 líneas (sin logs o con System.out.println)
TurnoService.java       ~80 líneas (con System.out.println mezclado)
Diagnóstico en prod     Imposible

DESPUÉS
ServiceLoggingDecorator ~40 líneas (cubre TODOS los servicios)
UserService.java        Sin ninguna línea de log — solo lógica de negocio
TurnoService.java       Sin ninguna línea de log — solo lógica de negocio
Diagnóstico en prod     Completo: servicio, método, args, duración, error
```

Reducción de código de logging distribuido en servicios: **100%**

### 6.3 Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Pointcut demasiado amplio | Baja | Bajo | Ajustar el patrón `execution(*)` al paquete correcto |
| AOP no disponible | Muy baja | Medio | spring-boot-starter-aop ya incluido indirectamente en Spring Boot |

### 6.4 Plan de Implementación

**Fase 1:**
- Agregar dependencia `spring-boot-starter-aop` en `pom.xml` de ambos microservicios

**Fase 2:**
- Crear `ServiceLoggingDecorator.java` con el pointcut apuntando al paquete correcto

**Fase 3:**
- Eliminar `System.out.println()` existentes en `TurnoService`
- Verificar en consola que los logs aparecen correctamente

---

 **Decisión Final**

Se aprueba la implementación del patrón Decorator via Spring AOP para agregar logging automático a todos los servicios del sistema, garantizando trazabilidad completa en producción sin modificar ninguna clase de lógica de negocio existente.
