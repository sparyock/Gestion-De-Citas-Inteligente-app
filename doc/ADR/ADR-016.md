## 2. Análisis del Proyecto

### 2.1 Estado Actual de la Arquitectura

El método `obtenerHorariosDisponibles()` devuelve siempre el mismo bloque de horas sin importar la especialidad. El método `obtenerDoctoresPorEspecialidad()` usa un `switch` estático. Cualquier nueva especialidad requiere modificar `TurnoService`, violando el principio Open/Closed.
```
┌─────────────────────────────────────────────┐
│            ARQUITECTURA ACTUAL              │
├─────────────────────────────────────────────┤

TurnoService
├── obtenerHorariosDisponibles(doctor)
│   └── return Lista fija hardcodeada (TODOS igual)
│
└── obtenerDoctoresPorEspecialidad(especialidad)
    └── switch (medicina general / cardiologia / ...) ← ESTÁTICO

└─────────────────────────────────────────────┘
```

**Problema detectado:** La lógica de disponibilidad está acoplada al servicio principal. Agregar una especialidad implica modificar código existente.

### 2.2 ADR Seleccionado

| ADR ID | Título | Prioridad | Razón |
|--------|--------|-----------|-------|
| ADR-016 | Strategy: Horarios por especialidad |  Alta | Eliminar switch estático y horas hardcodeadas en TurnoService |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Magic Code + Switch Gigante

**Ubicación:** `TurnoService.java`
```java
public List<String> obtenerHorariosDisponibles(String doctor) {
    return List.of("08:00","09:00","10:00","11:00","14:00","15:00","16:00");
    // Todos los doctores, todas las especialidades → mismo resultado
}
```

**Problemas:**
- Todos los doctores retornan exactamente los mismos horarios
- No hay diferenciación por especialidad ni por doctor
- Imposible extender sin modificar el método

### 3.2 Antipatrón: Switch Estático

**Ubicación:** `TurnoService.java`
```java
public List<String> obtenerDoctoresPorEspecialidad(String especialidad) {
    switch (especialidad.toLowerCase()) {
        case "medicina general": return List.of("Dr. Andrés Muñoz","Dra. Laura Soto");
        case "cardiologia":      return List.of("Dr. Felipe Ruiz","Dra. Laura Soto");
        case "dermatologia":     return List.of("Dr. Felipe Ruiz");
        case "pediatria":        return List.of("Dra. Camila Torres");
        default:                 return List.of();
    }
}
```

**Problemas:**
- Agregar una especialidad = editar este método
- Viola el principio Open/Closed
- Difícil de testear de forma aislada

## 4. Segmentos de Código con Malas Prácticas

### Lógica de disponibilidad sin polimorfismo
```java
// TurnoService.java — ANTES
// Un solo método para TODOS los doctores y especialidades
public List<String> obtenerHorariosDisponibles(String doctor) {
    return List.of("08:00","09:00","10:00","11:00","14:00","15:00","16:00");
}
```

**Problemas:**
- No hay diferencia entre especialidades (Pediatría debería tener solo mañanas)
- Cambiar los horarios de una especialidad afecta a todas
- No es posible agregar lógica de negocio específica sin romper las demás

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El sistema atiende múltiples especialidades médicas:
- Medicina General
- Cardiología
- Dermatología
- Pediatría

Cada especialidad tiene doctores distintos y rangos horarios distintos. La lógica de disponibilidad debe ser intercambiable en tiempo de ejecución.

### 5.2 Decisión

Se decide implementar el **patrón Strategy** encapsulando cada algoritmo de disponibilidad en una clase separada, inyectada automáticamente por Spring mediante un mapa de estrategias.

| Componente | Responsabilidad |
|------------|-----------------|
| `DisponibilidadStrategy` | Interfaz común de todas las estrategias |
| `MedicinaGeneralStrategy` | Horarios y doctores de Medicina General |
| `PediatriaStrategy` | Horarios solo de mañana + doctores de Pediatría |
| `TurnoService` | Delega en la estrategia correspondiente según especialidad |

### 5.3 Diseño de la Solución
```
┌────────────────────────────────────────────┐
│           ARQUITECTURA PROPUESTA           │
├────────────────────────────────────────────┤

        DisponibilidadStrategy (interface)
           getHorarios()
           getDoctores()

              ▲               ▲
              │               │

  MedicinaGeneralStrategy   PediatriaStrategy
  CardiologiaStrategy       DermatologiaStrategy

              │
              ▼

        TurnoService
        strategies.get(especialidad).getHorarios()

              │
              ▼

        TurnoController

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

**Interfaz Strategy**
```java
// DisponibilidadStrategy.java
public interface DisponibilidadStrategy {
    List<String> getHorarios();
    List<String> getDoctores();
}
```

**Estrategia concreta: Medicina General**
```java
// MedicinaGeneralStrategy.java
@Component("medicina general")
public class MedicinaGeneralStrategy implements DisponibilidadStrategy {

    public List<String> getHorarios() {
        return List.of("08:00","09:00","10:00","11:00","14:00","15:00","16:00");
    }

    public List<String> getDoctores() {
        return List.of("Dr. Andrés Muñoz", "Dra. Laura Soto");
    }
}
```

**Estrategia concreta: Pediatría (solo mañanas)**
```java
// PediatriaStrategy.java
@Component("pediatria")
public class PediatriaStrategy implements DisponibilidadStrategy {

    public List<String> getHorarios() {
        return List.of("08:00","09:00","10:00","11:00");
    }

    public List<String> getDoctores() {
        return List.of("Dra. Camila Torres");
    }
}
```

**TurnoService refactorizado**
```java
// TurnoService.java — REFACTORIZADO
@Service
public class TurnoService {

    // Spring inyecta automáticamente todas las implementaciones de
    // DisponibilidadStrategy usando el nombre del @Component como clave
    private final Map<String, DisponibilidadStrategy> strategies;

    public TurnoService(Map<String, DisponibilidadStrategy> strategies) {
        this.strategies = strategies;
    }

    public List<String> obtenerHorariosDisponibles(String especialidad) {
        return getStrategy(especialidad).getHorarios();
    }

    public List<String> obtenerDoctoresPorEspecialidad(String especialidad) {
        return getStrategy(especialidad).getDoctores();
    }

    private DisponibilidadStrategy getStrategy(String especialidad) {
        return strategies.getOrDefault(
            especialidad.toLowerCase(),
            strategies.get("medicina general") // fallback
        );
    }
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|-------------|---------|
| Open/Closed | Agregar especialidad = crear clase nueva, sin tocar TurnoService | Alto |
| Testabilidad | Cada estrategia se prueba con JUnit sin levantar contexto Spring | Alto |
| Legibilidad | El switch de 20 líneas desaparece del servicio principal | Medio |
| Extensibilidad | Nuevas especialidades se registran como @Component | Alto |

### 6.2 Métricas Esperadas
```
ANTES
TurnoService.java     ~90 líneas (switch + lógica mezclada)

DESPUÉS
DisponibilidadStrategy.java   ~5 líneas (interfaz)
MedicinaGeneralStrategy.java  ~15 líneas
PediatriaStrategy.java        ~15 líneas
TurnoService.java             ~40 líneas (sin switch)
```

Reducción de lógica condicional en el servicio principal: **~60%**

### 6.3 Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Especialidad sin estrategia registrada | Baja | Bajo | Fallback a "medicina general" implementado |
| Nombre de @Component con typo | Baja | Medio | Constantes para los nombres de especialidad |

### 6.4 Plan de Implementación

**Fase 1:**
- Crear interfaz `DisponibilidadStrategy`
- Crear `MedicinaGeneralStrategy` y `PediatriaStrategy`

**Fase 2:**
- Refactorizar `TurnoService` para usar mapa de estrategias

**Fase 3:**
- Agregar estrategias para Cardiología y Dermatología

---

 **Decisión Final**

Se aprueba la implementación del patrón Strategy para eliminar el switch estático y las horas hardcodeadas en `TurnoService`, permitiendo agregar nuevas especialidades sin modificar código existente.
