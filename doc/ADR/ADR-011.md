## 1. Identificación de Patrones

### 1.1 Patrones Actualmente Implementados

| Patrón               | Ubicación                       | Descripción                                        |
| -------------------- | ------------------------------- | -------------------------------------------------- |
| Repository Pattern   | UserRepository                  | Abstracción de acceso a datos usando JpaRepository |
| DTO Pattern          | UserRequestDTO, UserResponseDTO | Transferencia de datos entre capas                 |
| Service Layer        | UserService                     | Capa de lógica de negocio separada                 |
| Dependency Injection | @Autowired                      | Inyección de dependencias mediante Spring          |

### 1.2 Patrones Ausentes Críticos

| Patrón Faltante            | Justificación                                                        |
| -------------------------- | -------------------------------------------------------------------- |
| Interface + Implementación | UserService es una clase concreta, no implementa ninguna interfaz    |
| Dependency Inversion (DIP) | El controlador depende directamente de la clase concreta UserService |

## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado Actual de la Arquitectura

Actualmente el microservicio `users-service` tiene la siguiente estructura:

```
┌─────────────────────────────────────────────┐
│              ARQUITECTURA ACTUAL            │
├─────────────────────────────────────────────┤

Controller Layer
└── UserController
      │
      ▼
Service Layer (SIN INTERFAZ)
└── UserService (clase concreta)
      │
      ▼
Repository Layer
└── UserRepository

└─────────────────────────────────────────────┘
```

**Problema detectado:**

El `UserController` inyecta directamente `UserService` como clase concreta, violando el principio de inversión de dependencias.

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Concrete Class Dependency

**Ubicación:** `UserController.java`

```java
@Autowired
private UserService userService; // clase concreta directa
```

**Problema:**

- El controlador depende de la implementación, no de una abstracción
- Dificulta las pruebas unitarias
- Viola el principio de inversión de dependencias (DIP)

---

### 3.2 Antipatrón: Hard Dependency

**Ubicación:** `UserController.java`

```java
@RestController
@RequestMapping("/users")
public class UserController {

    @Autowired
    private UserService userService; // acoplado a la clase concreta
}
```

**Problemas:**

- Si se cambia UserService por otra implementación, hay que modificar el controlador
- No se puede mockear fácilmente en pruebas
- Rompe el principio Open/Closed

---

## 4. Segmentos de Código con Malas Prácticas

**Dependencia directa a clase concreta**

```java
// ❌ ANTES — acoplado a la implementación concreta
@Autowired
private UserService userService;
```

**Problemas:**

- No sigue el principio de inversión de dependencias
- Imposible cambiar la implementación sin tocar el controlador
- Las pruebas unitarias requieren instanciar toda la clase concreta

---

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El microservicio `users-service` maneja toda la lógica de usuarios. El controlador depende directamente de la clase de servicio concreta, generando acoplamiento fuerte entre capas.

---

### 5.2 Decisión

Se decide implementar el patrón Interface + Implementación, creando una interfaz `IUserService` que define el contrato, y `UserServiceImpl` que lo implementa.

| Componente      | Responsabilidad                   |
| --------------- | --------------------------------- |
| IUserService    | Define el contrato de operaciones |
| UserServiceImpl | Implementa la lógica de negocio   |
| UserController  | Depende solo de la interfaz       |

---

### 5.3 Diseño de la Solución

```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

        IUserService (interfaz)

              ▲
              │ implementa

       UserServiceImpl

              ▲
              │ inyecta

        UserController

└────────────────────────────────────────────┘
```

---

### 5.4 Implementación Propuesta

**Interfaz IUserService**

```java
public interface IUserService {
    UserResponseDTO create(UserRequestDTO request);
    List<UserResponseDTO> findAll();
    UserResponseDTO findById(Long id);
    UserResponseDTO update(Long id, UserRequestDTO request);
    void delete(Long id);
    UserResponseDTO login(String email, String password);
}
```

**Implementación UserServiceImpl**

```java
@Service
public class UserServiceImpl implements IUserService {

    @Autowired
    private UserRepository userRepository;

    @Override
    public UserResponseDTO create(UserRequestDTO request) {
        User user = new User();
        user.setNombre(request.getNombre());
        user.setEmail(request.getEmail());
        user.setPassword(request.getPassword());
        User saved = userRepository.save(user);
        return new UserResponseDTO(saved.getId(), saved.getNombre(), saved.getEmail());
    }
    // ... resto de métodos implementados
}
```

**Controlador usando la interfaz**

```java
// ✅ DESPUÉS — depende de la interfaz
@Autowired
private IUserService userService;
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio              | Descripción                                                   | Impacto |
| ---------------------- | ------------------------------------------------------------- | ------- |
| Desacoplamiento        | El controlador no depende de la implementación                | Alto    |
| Testabilidad           | Se puede mockear la interfaz en pruebas unitarias             | Alto    |
| Flexibilidad           | Se puede cambiar la implementación sin tocar el controlador   | Alto    |
| Buenas prácticas SOLID | Cumple el principio de inversión de dependencias              | Medio   |

---

### 6.2 Métricas Esperadas

```
ANTES

UserController → depende de UserService (clase concreta)
Pruebas        → requieren instanciar UserService completo


DESPUÉS

UserController → depende de IUserService (interfaz)
Pruebas        → mockean IUserService fácilmente
```

- Reducción de acoplamiento entre capas: Alta
- Mejora en testabilidad: Alta

---

### 6.3 Riesgos

| Riesgo                    | Probabilidad | Impacto | Mitigación                            |
| ------------------------- | ------------ | ------- | ------------------------------------- |
| Confusión con los nombres | Baja         | Bajo    | Convención clara: I + NombreServicio  |
| Refactorización inicial   | Media        | Bajo    | Cambio no rompe funcionalidad existente |

---

### 6.4 Plan de Implementación

**Fase 1:**
- Crear interfaz IUserService con todos los métodos

**Fase 2:**
- Renombrar UserService a UserServiceImpl
- Implementar IUserService en UserServiceImpl

**Fase 3:**
- Actualizar UserController para inyectar IUserService

---

✅ **Decisión Final**

Se aprueba la implementación del patrón Interface + Implementación en el microservicio `users-service` para desacoplar el controlador de la implementación concreta y mejorar la testabilidad del sistema.
