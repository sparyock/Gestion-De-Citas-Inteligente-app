# 1. Identificación de Patrones

## 1.1 Patrones Actualmente Implementados

En el proyecto *Gestión de Citas Inteligente* se han identificado los siguientes patrones de diseño ya implementados dentro de la arquitectura de microservicios.

| Patrón | Ubicación | Descripción |
|--------|-----------|------------|
| Repository Pattern | TurnoRepository, UserRepository | Acceso a datos mediante JpaRepository |
| DTO Pattern | TurnoDTO, UserDTO | Transferencia de datos entre capas |
| Service Layer | TurnoService, UserService | Lógica de negocio centralizada |
| Dependency Injection | @Autowired, @RequiredArgsConstructor | Inyección de dependencias |
| REST API Pattern | TurnoController | Comunicación mediante endpoints REST |

Estos patrones permiten separar responsabilidades y mantener una arquitectura organizada.

---

## 1.2 Patrones Ausentes Críticos

Durante el análisis del sistema se identificó que aún no se utiliza un mecanismo centralizado para convertir entidades a DTO, lo cual genera duplicación de código.

| Patrón Faltante | Justificación |
|----------------|--------------|
| Mapper Pattern | Conversión manual repetitiva entre Entity y DTO |
| MapStruct | Automatizar la conversión de objetos |
| Centralización del mapeo | Reducir errores y duplicación de código |
| Separación de responsabilidades | Evitar lógica repetitiva en servicios |

# 3. Identificación de Antipatrones

## 3.1 Antipatrón: Mapeo Manual Repetitivo

**Ubicación:** 

T`urnoServiceImpl.java  `
`UserServiceImpl.java  `

Ejemplo de código:

```java
private TurnoDTO mapToDTO(Turno turno) {

    TurnoDTO dto = new TurnoDTO();

    dto.setId(turno.getId());
    dto.setFecha(turno.getFecha());
    dto.setHora(turno.getHora());
    dto.setDoctor(turno.getDoctor());
    dto.setPaciente(turno.getPaciente());

    return dto;
}
```

**Problemas identificados:**

- Código repetido  
- Difícil mantenimiento  
- Posibilidad de errores  
- Baja escalabilidad  

---

## 3.2 Antipatrón: Violación del principio DRY

Cada servicio implementa su propio método de conversión.

Esto genera:

- Código duplicado  
- Inconsistencia en el sistema  
- Mayor tiempo de desarrollo  

---

## 3.3 Antipatrón: Alta dependencia en la lógica de servicio

El servicio realiza múltiples responsabilidades:

- Lógica de negocio  
- Conversión de datos  
- Persistencia  

Esto viola el principio:

Single Responsibility Principle (SRP)

# 5. Diseño e Implementación del ADR

## 5.1 Contexto

El sistema de Gestión de Citas Inteligente maneja múltiples entidades:

- Usuario  
- Doctor  
- Turno  
- Especialidad  

Cada una requiere conversión entre:

Entity  
DTO  

Actualmente esta conversión se realiza manualmente en los servicios.

Esto provoca:

- Duplicación de código  
- Mayor probabilidad de errores  
- Baja mantenibilidad  
- Mayor tiempo de desarrollo  

---

## 5.2 Decisión

Se implementará el *Mapper Pattern utilizando MapStruct* para automatizar la conversión entre:

`Entity ↔ DTO`

Esto permitirá:

- Reducir código repetitivo  
- Mejorar mantenibilidad  
- Aumentar escalabilidad  
- Separar responsabilidades  

---

## 5.4 Implementación Propuesta

Interfaz Mapper:

```java
@Mapper(componentModel = "spring")
public interface TurnoMapper {

    TurnoDTO toDTO(Turno turno);

    Turno toEntity(TurnoDTO dto);

}

```
Uso en el servicio:

```java
Turno turno = mapper.toEntity(dto);
Turno saved = repository.save(turno);
return mapper.toDTO(saved);

```
