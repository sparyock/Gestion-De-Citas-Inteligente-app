## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado Actual de la Arquitectura

Actualmente el sistema de Gestión de Citas Inteligentes puede ejecutar operaciones directamente desde los controladores sin utilizar una capa de servicios para manejar la lógica de negocio.

El sistema tiene una arquitectura en capas compuesta por:

- Controllers  
- Repositories  
- Entities  
- DTOs  

# Diagrama de arquitectura actual:


```
┌─────────────────────────────────────────────┐
│ ARQUITECTURA ACTUAL │
├─────────────────────────────────────────────┤

Controller Layer
└── UserController

Contiene:

Validaciones
Lógica de negocio

Acceso a datos

│

Repository Layer
└── UserRepository

Base de Datos
└── Tabla Usuarios

└─────────────────────────────────────────────┘

```


**Problema detectado:**

El controlador mezcla responsabilidades al contener lógica de negocio y acceso a datos, lo que dificulta el mantenimiento y la escalabilidad del sistema.

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Business Logic in Controller

**Ubicación:** `UserController.java`

Actualmente el controlador ejecuta lógica de negocio directamente.

Ejemplo:

```java
@PostMapping("/crear")
public User crear(User user){

    return userRepository.save(user);

}
```

**Problema:**

- Mezcla de responsabilidades
- Dificultad para mantenimiento
- Código poco organizado

---
### 3.2 Antipatrón: Tight Coupling Between Layers

El controlador depende directamente del repositorio.

**Esto genera:**

- Dependencias fuertes entre capas
- Dificultad para realizar pruebas
- Baja reutilización del código

---
### 3.3 Antipatrón: Lack of Service Layer

El sistema no utiliza una capa intermedia para manejar la lógica del negocio.

**Esto genera:**

- Código duplicado
- Baja mantenibilidad
- Dificultad para escalar el sistema

---
## 4. Segmentos de Código con Malas Prácticas

Ejemplo actual:

```java
@PostMapping("/crear")
public User crear(User user){

    return userRepository.save(user);

}
```

**Problemas:**

- El controlador contiene lógica de negocio
- No existe separación de responsabilidades
- Dificulta mantenimiento del sistema

---
## 5. Diseño e Implementación del ADR
### 5.1 Contexto

El sistema maneja múltiples entidades:

- Usuarios
- Turnos
- Doctores
- Especialidades

Estas entidades requieren lógica de negocio para validar datos, procesar información y manejar operaciones antes de acceder a la base de datos.

---
### 5.2 Decisión

Se decide utilizar la anotación @Service para crear una capa de servicios encargada de manejar la lógica de negocio del sistema.

**Esto permitirá:**

- Separar responsabilidades
- Mejorar organización del código
- Facilitar mantenimiento del sistema

---
### 5.3 Diseño de la Solución

Arquitectura propuesta:
```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

Controller Layer
└── UserController

   │

Service Layer
└── UserService

Contiene:

- Lógica de negocio
- Validaciones
- Reglas del sistema

   │

Repository Layer
└── UserRepository

Base de Datos
└── Tabla Usuarios

└────────────────────────────────────────────┘
```
---
### 5.4 Implementación Propuesta

``` Java
@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository){
        this.userRepository = userRepository;
    }

    public User crear(User user){

        return userRepository.save(user);

    }

}
```
