## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Entity Exposure

**Ubicación:** `TurnoController.java`

```java
@PostMapping
public Turno crearTurno(@RequestBody TurnoRequestDTO dto) {
    return turnoService.crearTurno(dto); // retorna entidad directamente
}
```

**Problema:**

- La entidad JPA queda expuesta directamente en la API
- Si la entidad cambia, la API cambia automáticamente
- Se exponen campos internos que el cliente no debería ver

---

### 3.2 Antipatrón: Tight Coupling between API and DB

**Ubicación:** `TurnoController.java`

```java
@GetMapping
public List<Turno> obtenerTodosTurnos() {
    return turnoService.obtenerTodosTurnos(); // lista de entidades
}
```

**Problemas:**

- La API queda acoplada al modelo de base de datos
- Cambios en la entidad rompen el contrato de la API
- No hay control sobre qué datos se exponen al cliente

---

## 4. Segmentos de Código con Malas Prácticas

**Retorno directo de entidad**

```java
// ❌ ANTES — entidad expuesta directamente
@PostMapping
public Turno crearTurno(@RequestBody TurnoRequestDTO dto) {
    return turnoService.crearTurno(dto);
}
```

**Problemas:**

- El cliente recibe campos de auditoría interna como `fechaCreacion`
- Si se agrega un campo a la entidad, aparece automáticamente en la API
- No hay separación entre el modelo de datos y el contrato de la API

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El `TurnoController` retorna directamente entidades JPA en sus respuestas HTTP. Esto expone la estructura interna de la base de datos y acopla la API al modelo de datos.

---

### 5.2 Decisión

Se decide crear un `TurnoResponseDTO` que define exactamente qué campos se exponen en la API. El servicio mapea la entidad al DTO antes de retornar la respuesta.

| Componente      | Responsabilidad                           |
| --------------- | ----------------------------------------- |
| Turno           | Entidad JPA — estructura de la BD         |
| TurnoRequestDTO | Datos de entrada para crear/actualizar    |
| TurnoResponseDTO | Datos de salida expuestos en la API      |

---

### 5.3 Diseño de la Solución

```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

Cliente → POST /turnos (TurnoRequestDTO)
              │
              ▼
TurnoController → turnoService.crearTurno(dto)
              │
              ▼
TurnoService → mapea Turno → TurnoResponseDTO
              │
              ▼
TurnoController → return ResponseEntity<TurnoResponseDTO>
              │
              ▼
Cliente recibe → solo los campos definidos en el DTO

└────────────────────────────────────────────┘
```

---

### 5.4 Implementación Propuesta

**Nuevo TurnoResponseDTO**

```java
public class TurnoResponseDTO {

    private Long idTurno;
    private String especialidad;
    private String doctor;
    private String fechaHora;
    private String estado;

    public TurnoResponseDTO(Long idTurno, String especialidad,
                             String doctor, String fechaHora, String estado) {
        this.idTurno = idTurno;
        this.especialidad = especialidad;
        this.doctor = doctor;
        this.fechaHora = fechaHora;
        this.estado = estado;
    }

    // getters...
}
```

**TurnoService con mapeo a DTO**

```java
// ✅ DESPUÉS — retorna DTO en lugar de entidad
public TurnoResponseDTO crearTurno(TurnoRequestDTO dto) {
    Turno saved = turnoRepository.save(turno);
    return new TurnoResponseDTO(
        saved.getIdTurno(),
        saved.getEspecialidad(),
        saved.getDoctor(),
        saved.getFechaHora().toString(),
        saved.getEstado().name()
    );
}
```

**TurnoController con ResponseEntity**

```java
// ✅ DESPUÉS — respuesta tipada con DTO
@PostMapping
public ResponseEntity<TurnoResponseDTO> crearTurno(@RequestBody TurnoRequestDTO dto) {
    return ResponseEntity.status(HttpStatus.CREATED)
            .body(turnoService.crearTurno(dto));
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio                  | Descripción                                                  | Impacto |
| -------------------------- | ------------------------------------------------------------ | ------- |
| Desacoplamiento API-BD     | La API ya no depende de la estructura de la BD               | Alto    |
| Control de datos expuestos | Solo se muestran los campos necesarios al cliente            | Alto    |
| Estabilidad de la API      | Cambios en la entidad no rompen el contrato de la API        | Alto    |
| Consistencia               | Mismo patrón que ya usa el users-service                     | Medio   |

---

### 6.2 Métricas Esperadas

```
ANTES

API retorna → Turno (entidad completa con campos internos)
Campos expuestos → idTurno, idUsuario, especialidad, doctor,
                   fechaHora, estado, fechaCreacion (campo interno)


DESPUÉS

API retorna → TurnoResponseDTO (solo campos necesarios)
Campos expuestos → idTurno, especialidad, doctor, fechaHora, estado
```

- Control sobre datos expuestos: Ninguno → Total
- Acoplamiento API-BD: Alto → Bajo

---

### 6.3 Riesgos

| Riesgo                          | Probabilidad | Impacto | Mitigación                              |
| ------------------------------- | ------------ | ------- | --------------------------------------- |
| Frontend rompe por cambio de campos | Baja     | Medio   | El DTO mantiene los mismos campos clave |
| Olvidar mapear un campo         | Baja         | Bajo    | Revisar el DTO contra los requerimientos |

---

### 6.4 Plan de Implementación

**Fase 1:**
- Crear TurnoResponseDTO con los campos necesarios

**Fase 2:**
- Actualizar TurnoService.crearTurno() para retornar TurnoResponseDTO

**Fase 3:**
- Actualizar TurnoController para usar ResponseEntity<TurnoResponseDTO>

---

✅ **Decisión Final**

Se aprueba la implementación de `TurnoResponseDTO` en el `TurnoController` para desacoplar la API del modelo de base de datos y tener control total sobre los datos expuestos al cliente.
