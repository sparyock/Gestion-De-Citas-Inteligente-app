## 2. Análisis del Proyecto

### 2.1 Estado Actual de la Arquitectura

El método `crearTurno()` persiste directamente sin validar especialidad, disponibilidad del doctor ni conflictos de horario. Esto permite guardar datos inválidos o inconsistentes en la base de datos.
```
┌─────────────────────────────────────────────┐
│            ARQUITECTURA ACTUAL              │
├─────────────────────────────────────────────┤

TurnoController
        │
        ▼
TurnoService.crearTurno()
        │
        │  ← SIN VALIDACIONES
        │
        ▼
turnoRepository.save()  ← guarda cualquier dato

└─────────────────────────────────────────────┘
```

**Problema detectado:** El sistema persiste turnos sin validar especialidad, disponibilidad del doctor ni conflictos de horario.

### 2.2 ADR Seleccionado

| ADR ID | Título | Prioridad | Razón |
|--------|--------|-----------|-------|
| ADR-018 | Chain of Responsibility: Validación de turnos | 🔴 Alta | crearTurno() persiste sin ninguna validación de reglas de negocio |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Validation Spaghetti

**Ubicación:** `TurnoService.java`
```java
public Turno crearTurno(TurnoRequestDTO dto) {
    Turno turno = new Turno();
    turno.setEspecialidad(dto.getEspecialidad()); // puede ser nulo o inválido
    turno.setDoctor(dto.getDoctor());             // sin verificar disponibilidad
    turno.setFecha(dto.getFecha());
    turno.setHora(dto.getHora());                 // sin verificar conflictos
    turno.setEstado(EstadoTurno.PENDIENTE);
    turno.setUsuarioId(dto.getUsuarioId());
    return turnoRepository.save(turno);           // persiste sin validar nada
}
```

**Problemas:**
- No valida que la especialidad exista en el sistema
- No verifica si el doctor está disponible en ese horario
- No detecta conflictos de turnos para el mismo doctor y hora
- Datos inválidos llegan directamente a la base de datos

## 4. Segmentos de Código con Malas Prácticas

### crearTurno() sin reglas de negocio
```java
// TurnoService.java — ANTES
public Turno crearTurno(TurnoRequestDTO dto) {
    Turno turno = new Turno();
    turno.setEspecialidad(dto.getEspecialidad());
    turno.setDoctor(dto.getDoctor());
    turno.setFecha(dto.getFecha());
    turno.setHora(dto.getHora());
    turno.setEstado(EstadoTurno.PENDIENTE);
    turno.setUsuarioId(dto.getUsuarioId());
    return turnoRepository.save(turno);
}
```

**Problemas:**
- Cualquier especialidad se acepta, incluso las que no existen
- Dos turnos pueden quedar registrados para el mismo doctor en el mismo horario
- Si se agregan validaciones con `if-else`, el método se vuelve inmanejable

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

Al crear un turno se deben verificar múltiples reglas de negocio en orden:
1. La especialidad debe existir en el sistema
2. El doctor debe pertenecer a esa especialidad
3. La hora debe ser válida para esa especialidad
4. El doctor no debe tener otro turno en ese horario

### 5.2 Decisión

Se decide implementar el **patrón Chain of Responsibility**: cada regla de negocio es un eslabón independiente que valida su condición y pasa al siguiente si es válida, o lanza excepción si no lo es.

| Componente | Responsabilidad |
|------------|-----------------|
| `TurnoValidationHandler` | Clase abstracta base de cada eslabón |
| `EspecialidadValidationHandler` | Verifica que la especialidad sea válida |
| `ConflictoHorarioHandler` | Verifica que no haya conflicto de horario |
| `TurnoService` | Construye y ejecuta la cadena antes de persistir |

### 5.3 Diseño de la Solución
```
┌────────────────────────────────────────────┐
│           ARQUITECTURA PROPUESTA           │
├────────────────────────────────────────────┤

TurnoService.crearTurno()
        │
        ▼
EspecialidadValidationHandler
  ¿especialidad válida? → NO → RuntimeException
        │ SI
        ▼
ConflictoHorarioHandler
  ¿doctor disponible?   → NO → RuntimeException
        │ SI
        ▼
turnoRepository.save()  ← solo llega si todo es válido

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

**Clase abstracta base del eslabón**
```java
// TurnoValidationHandler.java
public abstract class TurnoValidationHandler {
    private TurnoValidationHandler siguiente;

    public TurnoValidationHandler setSiguiente(TurnoValidationHandler sig) {
        this.siguiente = sig;
        return sig;
    }

    public final void validar(TurnoRequestDTO dto) {
        doValidar(dto);                                     // lógica de este eslabón
        if (siguiente != null) siguiente.validar(dto);     // pasa al siguiente
    }

    protected abstract void doValidar(TurnoRequestDTO dto);
}
```

**Eslabón 1: Especialidad válida**
```java
// EspecialidadValidationHandler.java
@Component
public class EspecialidadValidationHandler extends TurnoValidationHandler {

    private static final List<String> VALIDAS =
        List.of("medicina general","cardiologia","dermatologia","pediatria");

    protected void doValidar(TurnoRequestDTO dto) {
        if (!VALIDAS.contains(dto.getEspecialidad().toLowerCase()))
            throw new RuntimeException("Especialidad no válida: " + dto.getEspecialidad());
    }
}
```

**Eslabón 2: Sin conflicto de horario**
```java
// ConflictoHorarioHandler.java
@Component
public class ConflictoHorarioHandler extends TurnoValidationHandler {

    private final TurnoRepository repo;

    protected void doValidar(TurnoRequestDTO dto) {
        boolean existe = repo.existsByDoctorAndFechaAndHoraAndEstadoNot(
            dto.getDoctor(), dto.getFecha(),
            dto.getHora(), EstadoTurno.CANCELADO);
        if (existe)
            throw new RuntimeException("El doctor ya tiene turno en ese horario");
    }
}
```

**TurnoService con la cadena construida**
```java
// TurnoService.java — REFACTORIZADO
@Service
public class TurnoService {
    private final TurnoValidationHandler cadenaValidacion;

    public TurnoService(EspecialidadValidationHandler v1,
                        ConflictoHorarioHandler v2) {
        v1.setSiguiente(v2);                   // construir cadena: v1 → v2
        this.cadenaValidacion = v1;
    }

    public Turno crearTurno(TurnoRequestDTO dto) {
        cadenaValidacion.validar(dto);          // recorre toda la cadena
        Turno turno = new Turno();
        // ... setters ...
        return turnoRepository.save(turno);
    }
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|-------------|---------|
| SRP | Cada regla de validación vive en su propia clase | Alto |
| Extensibilidad | Agregar una regla = crear una clase y añadirla a la cadena | Alto |
| Orden garantizado | Las validaciones siempre se ejecutan en el mismo orden | Alto |
| Mantenibilidad | crearTurno() solo persiste, no valida | Medio |

### 6.2 Métricas Esperadas
```
ANTES
TurnoService.crearTurno()     ~15 líneas (sin validaciones)
Errores en BD                 Alta probabilidad de datos inválidos

DESPUÉS
TurnoValidationHandler.java        ~20 líneas (base abstracta)
EspecialidadValidationHandler.java ~15 líneas
ConflictoHorarioHandler.java       ~15 líneas
TurnoService.crearTurno()          ~10 líneas (solo persiste)
```

### 6.3 Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Excepción no manejada | Baja | Bajo | GlobalExceptionHandler ya maneja y devuelve 400/404 |
| Orden incorrecto de eslabones | Baja | Medio | Orden definido en el constructor de TurnoService |

### 6.4 Plan de Implementación

**Fase 1:**
- Crear clase abstracta `TurnoValidationHandler`
- Crear `EspecialidadValidationHandler`

**Fase 2:**
- Crear `ConflictoHorarioHandler`

**Fase 3:**
- Refactorizar `TurnoService.crearTurno()` para usar la cadena

---

✅ **Decisión Final**

Se aprueba la implementación del patrón Chain of Responsibility para organizar las validaciones de negocio al crear un turno, separando cada regla en su propio eslabón y garantizando que solo datos válidos lleguen a la base de datos.
