## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado Actual de la Arquitectura

Actualmente el sistema de Gestión de Citas Inteligentes utiliza controladores que retornan directamente objetos sin controlar el estado HTTP.

El sistema tiene una arquitectura en capas compuesta por:

- Controllers  
- Services  
- Repositories  
- Entities  
- DTOs  

# Diagrama de arquitectura actual:

 ```
┌─────────────────────────────────────────────┐
│ ARQUITECTURA ACTUAL │
├─────────────────────────────────────────────┤

Controller Layer
├── TurnoController
│ └── return TurnoDTO
│
└── UserController
└── return UserDTO

Service Layer
├── TurnoService
└── UserService

Repository Layer
├── TurnoRepository
└── UserRepository

└─────────────────────────────────────────────┘
```

**Problema detectado:**

Los controladores retornan directamente los objetos sin manejar códigos de respuesta HTTP, lo que puede generar respuestas inconsistentes y dificultad en el manejo de errores.

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Direct Object Return

**Ubicación:** `UserController.java`

```java
@GetMapping("/{id}")
public UserDTO getUser(Long id){

    return userService.findById(id);

}
``` 

**Problema:**

- No se controla el código HTTP  
- Respuestas inconsistentes  
- Dificultad para manejar errores  

---

### 3.2 Antipatrón: Lack of HTTP Status Handling

Los endpoints siempre retornan datos sin verificar si el recurso existe.

Ejemplo: 
GET /usuarios/10

Si el usuario no existe, el sistema puede generar errores.

**Esto genera:**

- Falta de control de errores  
- Mala experiencia del usuario  
- Dificultad para depuración  

---

## 4. Segmentos de Código con Malas Prácticas

Ejemplo actual:

```java
@GetMapping("/{id}")
public UserDTO getUser(Long id){

    return userService.findById(id);

}
```

**Problemas:**

- No se controla el estado HTTP
- No se manejan excepciones
- No se valida si el recurso existe

## 5. Diseño e Implementación del ADR
### 5.1 Contexto

El sistema expone múltiples endpoints para:

- Usuarios
- Turnos
- Doctores
- Especialidades

Estos endpoints deben retornar respuestas HTTP adecuadas para indicar si la operación fue exitosa o si ocurrió un error.

---
### 5.2 Decisión

Se decide utilizar la clase ResponseEntity en todos los controladores del sistema para manejar:

- Códigos HTTP
- Mensajes de respuesta
- Respuestas estructuradas

---
### 5.3 Diseño de la Solución

Arquitectura propuesta:

```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

Controller Layer
├── TurnoController
│     └── ResponseEntity<TurnoDTO>
│
└── UserController
      └── ResponseEntity<UserDTO>

Service Layer
├── TurnoService
└── UserService

Repository Layer
├── TurnoRepository
└── UserRepository

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

``` Java
@GetMapping("/{id}")
public ResponseEntity<UserDTO> getUser(
        @PathVariable Long id){

    UserDTO user = userService.findById(id);

    return ResponseEntity.ok(user);

}
```
