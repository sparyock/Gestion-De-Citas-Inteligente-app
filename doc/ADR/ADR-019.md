## 2. Análisis del Proyecto

### 2.1 Estado Actual de la Arquitectura

Los archivos `application.properties` de `turnos-service` y `users-service` contienen credenciales hardcodeadas. Estas credenciales están expuestas en el repositorio de GitHub. No existe una clase centralizada que gestione la configuración como una única instancia.
```
┌─────────────────────────────────────────────┐
│            ARQUITECTURA ACTUAL              │
├─────────────────────────────────────────────┤

turnos-service/application.properties
├── spring.datasource.password=admin1234  ← EXPUESTO EN GIT
└── spring.datasource.username=postgres

users-service/application.properties
├── spring.datasource.password=admin1234  ← MISMO PROBLEMA
└── spring.datasource.username=postgres

Clases del sistema
└── @Value("${spring.datasource.url}") ← lecturas dispersas

└─────────────────────────────────────────────┘
```

**Problema detectado:** Credenciales hardcodeadas en el repositorio y ausencia de una clase Singleton que centralice la configuración del sistema.

### 2.2 ADR Seleccionado

| ADR ID | Título | Prioridad | Razón |
|--------|--------|-----------|-------|
| ADR-019 | Singleton: Configuración centralizada |  Alta | Credenciales expuestas en GitHub y configuración dispersa entre clases |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: Hardcoded Configuration

**Ubicación:** `application.properties` (ambos microservicios)
```properties
# ANTIPATRÓN: credenciales en texto plano en el repositorio
spring.datasource.url=jdbc:postgresql://localhost:5432/turnosdb
spring.datasource.username=postgres
spring.datasource.password=admin1234  # ← expuesto en GitHub!
```

**Problemas:**
- Credenciales visibles para cualquier persona con acceso al repositorio
- Cambiar la contraseña requiere modificar el archivo y hacer commit
- El mismo valor `admin1234` aparece en múltiples microservicios

### 3.2 Antipatrón: Configuración Dispersa

**Problemas:**
- Distintas clases leen configuración con `@Value` de forma independiente
- No hay garantía de que todas lean el mismo valor actualizado
- No existe una única instancia de configuración en el contexto

## 4. Segmentos de Código con Malas Prácticas
```properties
# turnos-service/application.properties — ANTES
spring.datasource.url=jdbc:postgresql://localhost:5432/turnosdb
spring.datasource.username=postgres
spring.datasource.password=admin1234

# users-service/application.properties — MISMO PROBLEMA
spring.datasource.url=jdbc:postgresql://localhost:5432/usersdb
spring.datasource.username=postgres
spring.datasource.password=admin1234
```

**Problemas:**
- Contraseña idéntica hardcodeada en dos microservicios distintos
- Expuesta en el historial de commits de GitHub
- No permite configuración diferente por ambiente (dev, prod)

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El sistema tiene dos microservicios que comparten parámetros de infraestructura (credenciales de BD, secret JWT, URL del gateway). Centralizar estos valores en una única clase Singleton garantiza consistencia y permite que las credenciales reales vengan de variables de entorno en vez del código.

### 5.2 Decisión

Se decide implementar el **patrón Singleton** mediante un `@Component` de Spring con `@ConfigurationProperties`. Spring garantiza que existe una única instancia en todo el contexto de la aplicación.

| Componente | Responsabilidad |
|------------|-----------------|
| `SystemConfig` | Bean Singleton que centraliza toda la configuración |
| `application.yml` | Lee valores desde variables de entorno con fallback para desarrollo |
| `JwtService` | Consume `SystemConfig` en lugar de leer `@Value` directamente |

### 5.3 Diseño de la Solución
```
┌────────────────────────────────────────────┐
│           ARQUITECTURA PROPUESTA           │
├────────────────────────────────────────────┤

Variables de entorno (Docker / OS)
  DB_URL, DB_USER, DB_PASS, JWT_SECRET
        │
        ▼
  application.yml
  app.db-url: ${DB_URL:localhost...}
        │
        ▼
    SystemConfig  ← @Component (Singleton)
    una sola instancia en todo el contexto
        │
        ├──→ JwtService
        ├──→ TurnoService
        └──→ UserService

└────────────────────────────────────────────┘
```

### 5.4 Implementación Propuesta

**SystemConfig — Singleton gestionado por Spring**
```java
// SystemConfig.java
@Component
@ConfigurationProperties(prefix = "app")
public class SystemConfig {

    // Spring garantiza que este bean es Singleton (una sola instancia)
    private String dbUrl;
    private String dbUsername;
    private String dbPassword;
    private String jwtSecret;
    private long   jwtExpirationMs;
    private String gatewayUrl;

    // Getters (inmutable después de inicialización)
    public String getDbUrl()           { return dbUrl; }
    public String getJwtSecret()       { return jwtSecret; }
    public long   getJwtExpirationMs() { return jwtExpirationMs; }
    public String getGatewayUrl()      { return gatewayUrl; }

    // Setters necesarios para @ConfigurationProperties
    public void setDbUrl(String v)           { this.dbUrl = v; }
    public void setDbUsername(String v)      { this.dbUsername = v; }
    public void setDbPassword(String v)      { this.dbPassword = v; }
    public void setJwtSecret(String v)       { this.jwtSecret = v; }
    public void setJwtExpirationMs(long v)   { this.jwtExpirationMs = v; }
    public void setGatewayUrl(String v)      { this.gatewayUrl = v; }
}
```

**application.yml corregido**
```yaml
# application.yml — CORRECTO (variables de entorno, nunca secretos en Git)
app:
  db-url:            ${DB_URL:jdbc:postgresql://localhost:5432/turnosdb}
  db-username:       ${DB_USER:postgres}
  db-password:       ${DB_PASS:changeme}
  jwt-secret:        ${JWT_SECRET:secreto-desarrollo}
  jwt-expiration-ms: 86400000
  gateway-url:       ${GATEWAY_URL:http://localhost:8080}
```

**Uso del Singleton en JwtService**
```java
// JwtService.java
@Service
public class JwtService {

    private final SystemConfig config; // misma instancia en todo el contexto

    public JwtService(SystemConfig config) { this.config = config; }

    public String generarToken(String email) {
        return Jwts.builder()
            .setSubject(email)
            .setExpiration(new Date(System.currentTimeMillis()
                + config.getJwtExpirationMs()))
            .signWith(Keys.hmacShaKeyFor(config.getJwtSecret().getBytes()))
            .compact();
    }
}
```

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio | Descripción | Impacto |
|-----------|-------------|---------|
| Seguridad | Credenciales reales en variables de entorno, nunca en el código | Alto |
| Singleton garantizado | Spring crea una sola instancia de SystemConfig en todo el contexto | Alto |
| DevOps | docker-compose.yml inyecta variables según ambiente (dev/prod) | Alto |
| Mantenimiento | Cambiar JWT secret = cambiar variable de entorno, sin recompilar | Medio |

### 6.2 Métricas Esperadas
```
ANTES
application.properties    Credenciales en texto plano en GitHub
@Value dispersos          Lecturas independientes en cada clase

DESPUÉS
SystemConfig.java         ~40 líneas (única fuente de verdad)
application.yml           Variables de entorno con fallback
Credenciales en GitHub    NINGUNA — solo placeholders como ${DB_PASS}
```

### 6.3 Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Variable de entorno no definida en producción | Media | Alto | Fallback en application.yml para desarrollo |
| Historial de Git con contraseñas antiguas | Alta | Alto | Rotar credenciales y usar git-secrets para prevenir futuros commits |

### 6.4 Plan de Implementación

**Fase 1:**
- Crear `SystemConfig.java` con `@ConfigurationProperties`
- Migrar `application.properties` a `application.yml` con variables de entorno

**Fase 2:**
- Refactorizar `JwtService` para usar `SystemConfig`

**Fase 3:**
- Configurar variables de entorno en `docker-compose.yml` para dev y prod

---

 **Decisión Final**

Se aprueba la implementación del patrón Singleton mediante `@ConfigurationProperties` para centralizar toda la configuración del sistema en una única instancia, eliminando credenciales hardcodeadas del repositorio y garantizando consistencia entre todos los componentes.
