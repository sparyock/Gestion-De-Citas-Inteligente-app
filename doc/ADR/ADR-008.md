## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado Actual de la Arquitectura

Actualmente el sistema de Gestión de Citas Inteligentes permite listar todos los turnos registrados, pero no permite aplicar filtros en las consultas.

El sistema tiene una arquitectura en capas compuesta por:

- Controllers  
- Services  
- Repositories  
- Entities  
- DTOs  

# Diagrama de arquitectura actual:

```
┌─────────────────────────────────────────────┐
│ ARQUITECTURA ACTUAL │
├─────────────────────────────────────────────┤

Cliente

│

GET /turnos

│

Controller Layer
└── TurnoController

Service Layer
└── TurnoService

Repository Layer
└── TurnoRepository

Base de Datos
└── Tabla Turnos

Resultado:
Se retornan todos los registros

└─────────────────────────────────────────────┘
```

**Problema detectado:**

El sistema siempre retorna todos los registros, lo que puede afectar el rendimiento y dificulta la búsqueda de información específica.

## 3. Identificación de Antipatrones

### 3.1 Antipatrón: No Filtering Support

**Ubicación:** `TurnoController.java`

Actualmente el endpoint retorna todos los registros sin aplicar filtros.

Ejemplo:

```java
@GetMapping("/turnos")
public List<Turno> listar(){
    return turnoService.findAll();
}

```

**Problema:** 

- Consultas innecesarias
- Bajo rendimiento
- Dificultad para buscar datos específicos

---

### 3.2 Antipatrón: Large Data Response

El sistema retorna grandes cantidades de datos sin restricciones.

**Esto genera:**

- Alto consumo de memoria
- Lentitud en la respuesta
- Riesgo de sobrecarga del sistema

---

### 3.3 Antipatrón: Falta de Búsqueda Específica

El sistema no permite realizar búsquedas por criterios como:

- Fecha
- Doctor
- Especialidad
- Estado del turno

**Esto genera:**

- Pérdida de tiempo en consultas
- Dificultad para encontrar información
- Baja eficiencia en el sistema

### 4. Segmentos de Código con Malas Prácticas

Ejemplo actual:

```java
@GetMapping("/turnos")
public List<Turno> listar(){

    return turnoService.findAll();

}

```
**Problemas:**

- No permite filtros
- Retorna todos los registros
- Baja eficiencia en consultas
## 5. Diseño e Implementación del ADR
### 5.1 Contexto

El sistema maneja múltiples entidades:

- Turnos
- Usuarios
- Doctores
- Especialidades

Los usuarios necesitan buscar información específica sin tener que revisar todos los registros.

---

### 5.2 Decisión

Se decide utilizar la anotación @RequestParam para permitir el uso de parámetros en la URL y así filtrar resultados en los endpoints del sistema.

**Esto permitirá:**

- Realizar búsquedas específicas
- Mejorar el rendimiento del sistema
- Facilitar el uso del sistema

---

### 5.3 Diseño de la Solución

Arquitectura propuesta:

```
┌────────────────────────────────────────────┐
│            ARQUITECTURA PROPUESTA          │
├────────────────────────────────────────────┤

Cliente

   │

GET /turnos?fecha=2026-03-24

   │

Controller Layer
└── TurnoController

Service Layer
└── TurnoService

Repository Layer
└── TurnoRepository

Base de Datos
└── Tabla Turnos

Resultado:
Se retornan registros filtrados

└────────────────────────────────────────────┘
```

---
### 5.4 Implementación Propuesta

```Java
@GetMapping("/turnos")
public List<Turno> buscarPorFecha(
        @RequestParam String fecha){

    return turnoService.buscarPorFecha(fecha);

}
```
